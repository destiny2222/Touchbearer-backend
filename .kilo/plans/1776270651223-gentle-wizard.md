# Plan: Student Promotion & Academic Session-Based Results

## Problem Analysis

### Current System Behavior

1. **Students Table**: Has a single `class_id` field that points to the current class
2. **Terms Table**: Contains `name` (e.g., "First Term"), `session` (e.g., "2025/2026"), and `is_active`
3. **Results Table (`student_results`)**: Already stores:
   - `student_id` - the student
   - `class_id` - the class at the time result was saved (ALREADY CORRECT!)
   - `term_id` - the term when result was recorded
   - Assessment scores (ca1, ca2, ca3, ca4, exam)

4. **The Bug** is in **query logic**, NOT data storage:
   - In `/api/results/me/report-card` (line 713): filters by `sr.class_id = studentData.class_id`
   - This uses the student's CURRENT class, not the class stored in the result

### Key Discovery - Existing Data is Good!

The `student_results` table already stores the `class_id` at the time the result was saved. This is correct! The issue is just in how we **query** the results.

**Example:**
- Student in "JSS1" during "First Term 2025/2026" - results saved with class_id = JSS1 (CORRECT)
- Student promoted to "JSS2" for "First Term 2026/2027"
- When querying "First Term 2025/2026", we filter by JSS2 (current class) - WRONG!
- Should NOT filter by class at all, or use the class_id stored in the result

---

## Solution Design - MINIMAL CHANGES

### Strategy: Leverage Existing Data

Since `student_results` already stores `class_id` at the time of saving, we can:
1. **Fix the query logic** - Remove/change class filter when fetching past results
2. **No database changes needed** - The data is already correct!
3. **Promotion validation** - Ensure cross-session promotions work correctly

### 1. NO Database Changes Required

The existing `student_results.class_id` stores the correct historical class. We just need to fix how we query it.

### 2. Implementation Steps

#### Fix Results Query (`routes/results.js`)

**Report Card Endpoint** (`/api/results/me/report-card`):
- Current (WRONG): `WHERE sr.class_id = ? AND sr.term_id = ?` using student's current class_id
- Fixed: Remove class_id filter entirely, or query by student_id + term_id only

```javascript
// OLD (line 713):
let resultsQuery = `
    SELECT sr.student_id, sr.subject_id, cs.name as subject_name, sr.school_type, sr.assessment_type, sr.score
    FROM student_results sr
    JOIN class_subjects cs ON sr.subject_id = cs.id
    WHERE sr.class_id = ? AND sr.term_id = ? AND sr.published = TRUE
`;

// FIXED - Use student's historical class from the result itself:
let resultsQuery = `
    SELECT sr.student_id, sr.subject_id, cs.name as subject_name, sr.school_type, sr.assessment_type, sr.score
    FROM student_results sr
    JOIN class_subjects cs ON sr.subject_id = cs.id
    WHERE sr.student_id = ? AND sr.term_id = ? AND sr.published = TRUE
`;
// Note: Removed sr.class_id filter - results already have correct class_id stored!
```

**Same logic applies to**:
- `/api/results/student/:student_id` - Remove class_id filter
- Any other endpoint that filters results by student's current class

#### Step 2: Update Promotion Route (`routes/promote.js`)

Add validation to ensure cross-session promotion:
- When promoting, verify the target term is in a DIFFERENT session
- This ensures students can't be in two different classes within same session

```javascript
// Validate: Cross-session promotion
const currentTerm = getActiveTerm(); // First Term 2025/2026
const targetClass = getClass(next_class_id);
const targetTerm = getTermForClass(targetClass); // First Term 2026/2027

if (currentTerm.session === targetTerm.session) {
    // Same session - allow but warn (should use term promotion instead)
} else {
    // Different session - this is year promotion, record it
}
```
